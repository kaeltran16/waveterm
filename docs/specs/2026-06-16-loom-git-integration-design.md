# loom Git Integration — Design Spec

**Date:** 2026-06-16
**Status:** Design approved (brainstorm complete); implementation plan pending
**Base:** Extends the session sidebar (Phases 1–3). Replaces the Phase 3 diff-split (`Cmd:Shift:g` → `git --no-pager diff`) with a launch of the `loom` TUI git client, scoped to the active session's repo.
**Scope:** edits to `frontend/app/store/keymodel.ts`; one new typed config key in `pkg/wconfig` (`settingsconfig.go`, `metaconsts.go`, `defaultconfig/settings.json`); regenerated `frontend/types/gotypes.d.ts`. No new files, no new RPCs, no Go git package.

## 1. What this is

Today `Cmd:Shift:g` (`handleDiffSplit`, `keymodel.ts:405`) splits a non-interactive term block running `git --no-pager diff` in the active session's cwd — a flat, read-only diff.

This change repoints that same keybinding at **loom** (`C:\Users\kael02\IdeaProjects\loom`), a keyboard-driven TUI git client that already provides a changed-files list, stage/unstage (`space`), discard (`d`, confirm `y`), commit (`c`), branch switch, and fetch/pull/push. loom shells out to the real `git`, so it inherits the user's credentials, hooks, and config.

This is the **thin integration** (Approach A): Wave's core competency is hosting TUIs, so loom runs as an ordinary interactive term block. No diff rendering is reimplemented in Wave; loom is the single source of truth for the git-review experience, improved in one place.

## 2. Goals / Non-goals

**Goals**
- `Cmd:Shift:g`, in the active session, opens loom in a horizontal split scoped to that session's repo (its terminal cwd).
- The loom binary is resolved from a typed config key, defaulting to `loom` on `PATH`, so it works with zero config and is overridable for a non-PATH install.
- loom runs interactively (altscreen, keyboard) like the existing `claude`/`codex` agent sessions.

**Non-goals (this cut)**
- **No per-row context-menu entry point.** `Cmd:Shift:g` on the active session is the only trigger. Opening loom for a non-active session (switch-then-split, cross-tab timing) is explicitly deferred.
- **No Wave-native diff UI** (Monaco view, accept/reject toolbar, new git RPCs) — that was Approach C, dropped in favor of reusing loom.
- **No work in the loom repo.** loom's pending rich diff-panel spec (full-row backgrounds, word/syntax highlighting) is independent; loom's current flat diff is accepted as-is. Finishing it is a separate loom-side effort.
- **No bundling of the loom binary** into Wave's resources. PATH (or the config override) is the install contract.

## 3. The change (`keymodel.ts`)

Rename `handleDiffSplit` → `handleGitSplit` and rebuild its `blockDef` to launch loom. The function keeps its structure: resolve the active session's terminal block via the existing `findActiveSessionTermBlock()` (early-return no-op when there is none), then split a term block in after it.

```ts
async function handleGitSplit() {
    const termBlock = findActiveSessionTermBlock();
    if (termBlock == null) {
        return;
    }
    const loomBin = resolveLoomBin(); // app:loombin config, falls back to "loom"
    const blockDef: BlockDef = {
        meta: {
            view: "term",
            controller: "cmd",
            cmd: loomBin,
            "cmd:interactive": true, // loom is an altscreen TUI, not a run-once command
            "cmd:cwd": termBlock.cwd,
        },
    };
    await createBlockSplitHorizontally(blockDef, termBlock.blockId, "after");
}
```

Notes:
- The `cmd:interactive: true` flag mirrors how the sidebar already launches interactive agents (`buildDuplicateBlockMeta`'s `{ controller: "cmd", cmd: "claude", "cmd:interactive": true }`). It replaces the previous non-interactive `git --no-pager diff` (whose pager-avoidance comment is removed with it).
- The `Cmd:Shift:g` binding in `registerGlobalKeys()` (`keymodel.ts:562`) is unchanged except for calling `handleGitSplit`.
- `resolveLoomBin()` reads the `app:loombin` config through Wave's existing settings accessor and falls back to `"loom"` when unset/empty. The exact accessor symbol is confirmed in the implementation plan against how other `app:*` settings are read on the frontend.

## 4. Config key — `app:loombin`

A single typed settings key, added the same way as `app:tabbar`:

- `pkg/wconfig/settingsconfig.go` — add to `SettingsType` (beside `AppTabBar` at :71):
  ```go
  AppLoomBin string `json:"app:loombin,omitempty"`
  ```
- `pkg/wconfig/metaconsts.go` — add the const (beside `ConfigKey_AppTabBar` at :20):
  ```go
  ConfigKey_AppLoomBin = "app:loombin"
  ```
- `frontend/types/gotypes.d.ts` — regenerated by `task generate` (never hand-edited); `app:loombin` becomes a typed config key.
- **Default:** `"loom"`, applied as a code-level fallback in `resolveLoomBin()` so the feature works with no config. A baked `"app:loombin": "loom"` in `defaultconfig/settings.json` is optional and redundant with the fallback; the fork owner may add it for visibility but it is not required.

This is the only backend touch: one struct field, one const. No git package, no RPCs.

## 5. Edge cases — loom handles them

- **cwd is not a git repo:** loom prints `run loom from inside a git repository` and exits; the term block shows that. No Wave-side check needed.
- **Binary missing / misconfigured `app:loombin`:** the `cmd` controller fails to start and surfaces the OS error (e.g. command not found) in the block. Acceptable for a thin integration.
- **No active session / no terminal block:** the existing `findActiveSessionTermBlock() == null` guard no-ops, exactly as the diff-split did.

## 7. Testing / verification

No unit tests: this is keymap + config wiring, which has no unit coverage in this codebase (consistent with Phases 1–3, where keybindings and `wsh` integration were verified live). Verified by running the app:

1. With `app:tabbar=left`, open a session in a git repo; press `Cmd:Shift:g`. **Expect:** a horizontal split opens running loom, showing that repo's status.
2. In the loom split, stage (`space`), discard (`d`/`y`), and commit (`c`) — confirm they act on the session's repo.
3. Press `Cmd:Shift:g` in a session whose cwd is **not** a git repo. **Expect:** loom's "run loom from inside a git repository" message in the block (no crash).
4. Set `app:loombin` to an explicit path (e.g. the fork's `loom.exe`); reload; `Cmd:Shift:g` launches that binary. Set it to a bogus value; confirm the block shows a command-not-found error rather than failing silently elsewhere.
5. Press `Cmd:Shift:g` with no active terminal block. **Expect:** no-op.

## 8. Fork hygiene

The only stock-Wave files touched are `keymodel.ts` (repurposing one already-custom handler — the diff-split was a fork addition) and `pkg/wconfig` (two additive lines, mirroring `app:tabbar`). No rebinding of existing keys, no new files, no new dependencies. Consistent with the additive-over-edit principle from the session-sidebar specs; minimal upstream-rebase surface.
